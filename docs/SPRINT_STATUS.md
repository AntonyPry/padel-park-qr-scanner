# Статус спринтов CRM

Дата фиксации: 2026-05-22.

Этот файл - единый источник правды по спринтам. Перед началом нового спринта сначала обновляем этот документ: что уже сделано, что частично, что осталось, какие критерии приемки.

## Статусы

- `done` - задача закрыта как production-функциональность: backend, frontend, права, миграции, UX, проверки.
- `partial` - рабочий слой уже есть, но не закрыты все критерии production-версии.
- `todo` - функциональность еще не реализована.
- `blocked` - нельзя закрыть без внешнего решения или данных.

## Правило работы дальше

Новые задачи не должны создавать новый roadmap поверх старого. Если появляется новая потребность, она добавляется в существующий спринт или оформляется отдельным новым спринтом с явной причиной.

Перед стартом Sprint N:

1. Проверяем этот файл.
2. Отмечаем уже реализованное.
3. Формируем короткий план только по оставшимся пунктам.
4. После разработки обновляем статус и критерии проверки.

## Sprint 11 - Клиентская CRM 2.0

Цель: сделать карточку клиента главным рабочим инструментом менеджера.

Общий статус: `done`.

Уже реализовано:

- раздел клиентов с таблицей, фильтрами, сегментами, пагинацией и режимом дублей;
- карточка клиента с основной информацией, статистикой, визитами и тренировочными заметками;
- создание, редактирование, архивирование, восстановление и удаление архивного клиента;
- восстановление архивного клиента при попытке создать клиента с таким же телефоном;
- нормализация и маска телефона;
- защита дублей по телефону;
- объединение дублей по телефону с переносом истории;
- источники клиента через справочник;
- скрытие телефонов для роли `trainer` на уровне клиентского сервиса;
- базовые быстрые действия из карточки клиента;
- единая лента клиента в карточке: визиты, тренировки, обзвоны и изменения;
- сохраненные представления/фильтры клиентов на уровне аккаунта;
- расширенная проверка дублей и конфликтов по `telegramId`, `vkId`, `webId`;
- режим дублей показывает группы по телефону, Telegram, VK и WEB ID.
- из карточки клиента можно создать задачу обзвона без промежуточной базы;
- Telegram/VK/WEB ID можно редактировать в форме клиента;
- audit log редактирования клиента скрывает чувствительные поля перед записью;
- timeline клиента показывает список измененных полей без раскрытия телефона и внешних ID.
- direct-задача обзвона из карточки клиента не записывается в timeline как ложное "изменение клиента";
- проверена матрица ролей клиентского API: owner/manager/admin/viewer/trainer/accountant/unauthorized;
- проверен безопасный режим `trainer`: телефон, phoneNormalized и внешние ID скрыты, lookup по телефону запрещен;
- UI клиентов проверен в браузере: список, пагинация по 10 записей, форма клиента, карточка, модалка создания задачи, console без ошибок.

Осталось закрыть в рамках Sprint 11:

- критичных хвостов нет.

Отложено до отдельного подтвержденного запроса:

- экспорт/массовые действия по клиентам;
- diff старое/новое в timeline без раскрытия чувствительных данных.

Критерии приемки:

- менеджер открывает клиента и видит всю историю работы в одном месте;
- из карточки можно создать задачу, изменить статус, архивировать/восстановить клиента;
- тренер не видит телефон и лишние персональные данные ни в списке, ни в карточке, ни в API;
- дубли блокируются или объединяются предсказуемо по телефону и messenger/web id;
- фильтры можно сохранить и быстро открыть повторно.

- статусы клиента в задаче: new, callback, doubting, booked, refused, no_answer;

- архивирование, восстановление и удаление архивных задач/баз;
- контекстное меню действий по задаче;

- после одиночных и массовых действий обновляются список, карточка задачи, клиенты и отчет;

- критичных хвостов нет.

- отдельная аналитика по исполнителям и периодам;

- исполнитель видит список клиентов, дедлайны, статусы и историю попыток;
- руководитель видит прогресс, просрочки и результативность обзвона;
- архивные задачи можно восстановить или удалить навсегда с подтверждением.

## Sprint 13 - Тренерский кабинет

Цель: отдельный безопасный режим для тренеров.

Общий статус: `done`.

Уже реализовано:

- роль `trainer`;
- права на просмотр клиентов для тренера;
- запрет на управление клиентской базой для тренера через матрицу доступов;
- скрытие телефона и messenger id в клиентском сервисе для тренера;
- модель `TrainingNote`;
- API тренировочных заметок;
- дневник тренировок в карточке клиента;
- уровень клиента: `D`, `D+`, `C`, `C+`, `B`, `B+`, `A`;
- упражнения и свободные заметки;
- история тренировочных записей;
- отдельный раздел `/admin/trainer` для тренера, owner и manager;
- тренер по умолчанию попадает в тренерский кабинет, а не в общий раздел клиентов;
- тренерский кабинет показывает только безопасные клиентские данные без телефона, Telegram, VK и webId;
- API деталей клиента для trainer не отдает историю визитов, номера ключей, CRM-note и общую timeline;
- фильтр клиентов по последнему тренировочному уровню;
- удобная история прогресса по уровням в выбранной карточке;
- редактирование и удаление тренировочных записей: owner/manager могут менять все, trainer только свои;
- API `PUT /training-notes/:noteId` и `DELETE /training-notes/:noteId`;
- мобильный доступ к меню через отдельную кнопку открытия сайдбара;
- smoke-check на локальной БД и UI-проверка тренерского кабинета.

Решение по привязке:

- отдельная привязка клиента к конкретному тренеру пока не вводилась: текущая модель хранит автора каждой тренировочной записи;
- если бизнес-процесс потребует распределять клиентов по тренерам, это нужно вынести в отдельный спринт как назначение тренера/группы/расписания.

Критерии приемки:

- тренер логинится и видит только безопасный набор данных;
- тренер может добавить запись тренировки, уровень, упражнения и заметку;
- тренер не может создать, удалить, объединить или массово изменить клиента;
- телефон клиента не утекает ни в UI, ни в network/API ответах;
- менеджер/owner видит тренировочную историю в карточке клиента.

Цель: убрать хаос из статусов, источников, категорий и архивов.

Общий статус: `done`.

Уже реализовано:

- раздел справочников CRM;
- справочник источников клиента;
- справочник категорий визита;
- статусы справочников `active` / `archived`;

- использование источников клиента из справочника в форме клиента;
- использование категорий визита из справочника во входах;
- архив/восстановление/удаление для клиентов, баз, задач, сотрудников, аккаунтов, категорий каталога;
- подтверждения перед опасными действиями в основных местах;
- общий audit log по мутациям.
- единый `ConfirmActionDialog` для опасных действий в справочниках CRM, каталоге товаров, базах, задачах, клиентах, сотрудниках, аккаунтах, мотивации и тренировочных заметках;
- permanent delete задач обзвона разрешен только для архивного мусора без истории звонков, статусов и комментариев;
- permanent delete категорий каталога запрещен, если категория участвует в мотивации, финансовой истории или использованных в чеках правилах;
- базы с архивными клиентами можно хранить как фильтры, но нельзя использовать для задач обзвона и автозадач;
- архивные справочники не попадают в новые формы выбора, но старые значения продолжают отображаться в существующих записях;
- UI-проверка экранов `/admin/client-bases`, `/admin/catalog`, `/admin/references`, `/admin/motivation`;
- smoke-check на локальной БД: удаление архивной задачи без истории, запрет удаления задачи с историей, запрет удаления категории с финансовой историей, запрет автозадачи по архивным клиентам.

Осталось закрыть:

- критичных хвостов нет.

Отложено до отдельного подтвержденного запроса:

- архив/restore для самих правил мотивации как отдельная сущность вместо текущего удаления из списка;
- архивирование смен вместо удаления/корректировки payroll-истории. Это лучше закрывать в Sprint 15 вместе с workflow утверждения payroll.

Критерии приемки:

- во всех разделах понятно, где активные данные, где архив;
- любую архивируемую сущность можно восстановить, если это не ломает целостность данных;
- мусорные архивные сущности можно удалить навсегда только там, где нет связанной истории;
- старые записи не ломаются, если их источник/категория ушли в архив;
- опасные действия требуют подтверждения.

- UI финансов показывает финансовую историю человекочитаемыми действиями и не падает в пустой экран при ошибке загрузки;

- утвержденный payroll нельзя случайно изменить;
- P&L сходится с чеками и ручными операциями;
- спорное изменение можно найти в журнале.

Критерии приемки:
