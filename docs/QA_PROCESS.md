# QA-процесс после изменений

Дата фиксации: 2026-05-21.

Этот документ фиксирует обязательный способ проверки проекта после изменений. Цель - не просто "код собрался", а убедиться, что функциональность рабочая, безопасная, удобная и не выглядит как странное решение для реальной CRM.

## Definition of Done

Фича считается закрытой только если выполнены все применимые пункты:

- backend реализован и проверен;
- frontend реализован и проверен визуально;
- миграции применяются на локальной базе;
- есть тестовые или моковые данные для проверки сценария, если фича работает с данными;
- роли и доступы проверены;
- опасные действия требуют подтверждения;
- архив/восстановление/удаление продуманы, если сущность архивируемая;
- нет утечек телефона, токенов, паролей и других чувствительных данных;
- UI не ломается на реальных объемах данных;
- проверено, что решение не противоречит нормальной CRM-практике.

## Обязательный цикл проверки

### 1. Проверка здравого смысла

Перед реализацией или перед финальной сдачей:

- зачем эта функция нужна пользователю;
- кто ей пользуется: owner, manager, admin, accountant, trainer, viewer;
- не смешиваем ли мы разные сущности в одном разделе;
- не создаем ли хардкод там, где должен быть справочник или настройка;
- как это делают обычные CRM: список, карточка, история, статусы, фильтры, задачи;
- можно ли объяснить фичу заказчику в 2-3 предложениях.

Если запрос выглядит рискованно или странно для CRM, сначала фиксируем альтернативу и объясняем компромисс.

### 2. Данные для проверки

Если фича работает с данными, проверка без данных не считается достаточной.

Нужно подготовить:

- минимум 5-10 реалистичных записей для списков и таблиц;
- граничные кейсы: архивная запись, пустое значение, длинное имя, несколько связанных сущностей;
- конфликтный кейс: дубль, занятая категория, закрытая смена, запрещенная роль;
- если данные временные - использовать префикс `[qa]` или `codex-...` и удалить после проверки;
- если данные нужны пользователю для ручной проверки - оставить и явно написать, что создано.

Для data-driven фичей обязательно проверяем:

- создание;
- редактирование;
- архивирование;
- восстановление;
- удаление навсегда, если разрешено;
- фильтрацию;
- пагинацию;
- пустое состояние.

### 3. Backend

Минимум:

```bash
node --check <changed-backend-file.js>
git diff --check
```

Если есть миграции:

```bash
npx sequelize-cli db:migrate
```

Если менялись API:

- проверить happy path;
- проверить валидацию плохих данных;
- проверить 401 без токена;
- проверить 403 для роли без прав;
- проверить, что 500 не отдает внутренние stack/message наружу;
- проверить audit log для важных POST/PUT/PATCH/DELETE;
- проверить, что чувствительные поля редактируются в логах.

Для финансов, оплат и webhook:

- smoke payload на cash;
- smoke payload на cashless;
- smoke payload на mixed;
- проверка итогов в API и UI;
- проверка, что временные чеки удалены после теста, если они не нужны.

### 4. Frontend

Минимум:

```bash
npm run lint
npm run build
```

Если менялся UI, обязательно открыть приложение в браузере и проверить:

- нужный раздел загружается;
- нет ошибок в console;
- таблицы не вылезают за контейнер;
- в таблицах с реальными длинными строками текст не перекрывает соседние колонки;
- технические сокращения вроде SLA/API ID не выводятся пользователю без понятного пояснения или замены на рабочий термин;
- модалки нормального размера;
- кнопки и селекты читаемы;
- текст не обрезается бессмысленно;
- пустое состояние понятно;
- loading/error состояния не ломают экран;
- destructive action открывает подтверждение;
- после действия данные обновляются без ручной перезагрузки, если это ожидаемо.

### 5. Скриншоты нового фронтенда

Для новых или сильно измененных экранов нужно делать визуальную фиксацию.

Папка:

```text
outputs/qa/YYYY-MM-DD-feature-name/
```

Минимальный набор:

- desktop screenshot основного экрана;
- desktop screenshot модалки/формы;
- screenshot пустого состояния, если оно важно;
- screenshot состояния с большим количеством данных;
- mobile/narrow screenshot, если экраном реально могут пользоваться на узком окне.

В финальном отчете указывать пути к скриншотам, если они были сделаны.

### 6. UX-проверка

Проверяем не только "работает", но и "можно ли этим пользоваться".

Контрольные вопросы:

- понятно ли, что делать дальше;
- не спрятана ли главная кнопка;
- есть ли подтверждение перед опасным действием;
- можно ли отменить или восстановить ошибочное действие;
- не слишком ли много ручной работы;
- видно ли результат действия;
- не выглядит ли экран как debug/admin dump;
- поместятся ли реальные данные заказчика.

### 7. Проверка безопасности и ролей

Для каждой фичи определить роли:

- кто видит;
- кто создает;
- кто редактирует;
- кто архивирует;
- кто восстанавливает;
- кто удаляет навсегда;
- кто видит телефон и персональные данные;
- кто видит финансы.

Особые правила:

- `trainer` не должен видеть телефоны и лишние персональные данные;
- `viewer` не должен мутировать данные;
- `accountant` не должен управлять операционкой, если это не оговорено;
- destructive API должны быть закрыты на backend, а не только скрыты в UI;
- логи не должны содержать токены, пароли, authorization headers, телефоны, если это не осознанное debug-исключение.

### 8. Проверка на адекватность решения

Перед финальным ответом нужно отдельно оценить:

- соответствует ли решение задаче заказчика;
- не является ли фича временным костылем вместо нормальной модели;
- не захардкожены ли бизнес-правила, которые должен менять owner/manager;
- не создали ли мы новую сущность, которую нельзя администрировать;
- не забыты ли архив, восстановление и audit;
- не нужно ли вынести настройку в справочник.

Если решение сомнительное, в отчете нужно сказать об этом прямо и предложить более правильный вариант.

### 9. Когда использовать дополнительную проверку

Для крупных изменений нужно делать вторую независимую проверку:

- code review логики и безопасности;
- UI/UX review измененных экранов;
- smoke-тест пользовательского сценария от начала до конца.

Если в текущем режиме разрешены отдельные агенты, можно разделять:

- один агент - ревью backend/frontend логики;
- второй агент - браузерная проверка UI и функциональности.

Если агенты недоступны или не разрешены, эти проверки выполняются вручную в том же цикле.

### 10. Финальный отчет после разработки

В ответе пользователю нужно кратко указать:

- что изменено;
- какие миграции применены;
- какие тестовые данные созданы или удалены;
- какие проверки прошли;
- что не удалось проверить;
- известные ограничения или риски;
- где смотреть результат в интерфейсе.

## Минимальная матрица регрессии по модулям

### Клиенты

- создать клиента;
- проверить маску телефона;
- проверить дубль активного клиента;
- проверить восстановление архивного клиента;
- открыть карточку;
- проверить визиты;
- проверить training notes;
- архивировать/восстановить;
- удалить навсегда только архивного и только если нет истории.

### Базы и обзвоны

- создать базу;
- проверить, что база с архивными клиентами не дает создать задачу или автозадачу;
- создать задачу из базы;
- проверить snapshot/dynamic;
- изменить статус задачи;
- изменить статус клиента в задаче;
- добавить попытку звонка;
- поставить дедлайн;
- увидеть просрочку;
- архивировать/восстановить задачу.
- удалить навсегда только архивную задачу без статусов, комментариев и попыток звонка.

### Тренер

- войти тренером;
- открыть список клиентов;
- убедиться, что телефонов нет;
- добавить тренировочную запись;
- проверить, что нельзя управлять клиентской базой.

### Справочники

- создать источник/категорию;
- выбрать в форме клиента или визита;
- архивировать;
- убедиться, что в новых формах архивное не предлагается;
- восстановить;
- удалить навсегда, если нет связей.
- проверить, что категорию каталога нельзя удалить навсегда, если по ней есть финансы, мотивация или использованные в чеках правила.

### Финансы и мотивация

- проверить текущую смену;
- проверить продажи по категориям;
- проверить бонусы;
- проверить нал/безнал;
- закрыть смену;
- скопировать текстовый отчет;
- проверить payroll;
- проверить P&L.
- создать payroll-период;
- пересчитать черновик payroll;
- отправить payroll на проверку;
- убедиться, что смену в проверенном/утвержденном периоде нельзя изменить;
- убедиться, что ручную финансовую операцию в закрытом периоде нельзя добавить;
- утвердить payroll owner/manager;
- отметить payroll выплаченным owner/accountant;
- проверить финансовую историю;
- проверить XLSX-экспорт payroll;
- проверить XLSX-экспорт P&L;
- проверить сверку чеков и позиций Эвотора.

### Входы и сканер

- ручной вход;
- QR scan;
- модалка после скана;
- ключ;
- категория визита;
- повторный скан;
- disconnect scanner;
- reconnect scanner.
