# ТЗ: формат onboarding-инструкций CRM

## Цель

Сделать инструкции понятными для сотрудника, который впервые видит CRM: сначала он узнает раздел, затем читает пошаговое действие, затем сверяет результат.

Формат не использует режим `Попробовать`, quest bar, CSS-callouts или crop-фрагменты. Основной носитель смысла - текст шага. Скриншоты обычно нужны как спокойное подтверждение стартового экрана; для аналитических или сложных review-сценариев допускаются только вшитые в реальный CRM screenshot небольшие пронумерованные стрелки, если они явно согласованы в ТЗ и не закрывают контент.

## Базовая структура урока

Для release-quality уроков со скриншотами используется формат `section-first-cards`.

Карточка 1: `Открой ...`

- называет конкретный стартовый экран: `Открой раздел «Клиенты»`, `Открой карточку клиента`, `Открой раздел «Бронирования»` и т.д.;
- всегда показывает один реальный скриншот раздела, карточки или рабочего экрана CRM, с которого начинается сценарий;
- коротко просит открыть этот экран перед началом инструкции;
- обычно не содержит стрелок, рамок, номеров или сгенерированных иллюстраций; согласованные вшитые номера на реальном CRM screenshot должны иметь текстовую расшифровку под изображением.

Карточка 2: `Что нажать`

- текстом объясняет, как открыть нужную форму, карточку, модалку или действие;
- обычно без скриншота, если общий раздел уже показан на первой карточке;
- не использует тестовые ФИО, телефоны или training-данные как обязательные значения.

Карточка 3: `Что заполнить`

- объясняет, какие поля выбрать или заполнить;
- если есть форма или модалка, показывает один общий реальный скриншот формы;
- текст должен говорить, что заполнить универсально: имя клиента, телефон из обращения, источник, комментарий, дата, время, статус, сумма и т.д.

Карточка 4: `Как проверить результат`

- объясняет, по каким признакам пользователь понимает, что действие выполнено;
- если есть экран результата, показывает один общий реальный скриншот результата;
- проверка должна быть конкретной: запись появилась в списке, статус изменился, остаток обновился, бронь видна в расписании.

## Review и knowledge-уроки

Review-уроки и knowledge-гайды используют тот же принцип, если у них есть скриншот:

- первая карточка всегда называет стартовый экран командой `Открой ...`;
- последующие карточки могут быть текстовыми, если они объясняют метрики, формулы, статусы, жизненный цикл или правила доступа;
- скриншот добавляется только тогда, когда он прямо помогает понять конкретный экран или состояние.

Если у урока пока нет реального screenshot asset, он может оставаться текстовым до отдельного screenshot refresh. Нельзя заменять отсутствующий CRM-скриншот схемой или декоративной картинкой.

## Требования к скриншотам

- Только реальные CRM screenshots из локального приложения.
- Без CSS-callouts, crop-фрагментов, сгенерированных иллюстраций и декоративных рамок.
- Вшитые небольшие стрелки/номера допустимы только для сложных review-сценариев, если это реальные CRM screenshots, номера не закрывают контент, а `lesson.screenshots[].callouts` содержит расшифровку каждого номера.
- Синтетические demo CRM данные разрешены, если они выглядят как нормальный локальный демо-клуб и не являются персональными данными реальных клиентов.
- Не использовать странные QA-названия, случайные `QA DND ...`, `[training]` и обязательные тестовые ФИО в видимой части сценария.
- Demo-телефоны допустимы только как синтетические локальные данные на admin/accountant/owner/manager screenshots; для trainer-facing screenshots телефоны должны быть скрыты.
- Trainer-facing screenshots не должны раскрывать телефоны, внешние ID, историю звонков и лишние персональные данные.
- Каждый screenshot должен лежать в `client/public/onboarding/...` и быть связан через `lesson.screenshots`.

## Release gate

Перед QA:

- каждый урок должен иметь `updatedAt`: если урок обновился после `progress.completedAt`, пользователь видит пометку `Обновлено`;
- повторное завершение обновленной инструкции отмечает текущую версию прочитанной и убирает пометку без сброса прогресса;
- `server npm run onboarding:audit:strict` должен проходить без warnings;
- все screenshot-backed cards должны иметь существующий PNG;
- browser QA должен проверить `/admin/onboarding/:taskKey` на desktop и `390px`;
- console warning/error, horizontal overflow и переполнение нижних кнопок не допускаются.

Текущий baseline:

- `114/114` onboarding tasks имеют lesson;
- `99` уроков со скриншотами используют `section-first-cards`;
- `15` уроков пока текстовые без screenshot assets;
- `157/157` screenshot-backed cards закрыты реальными PNG;
- `698` карточек намеренно текстовые.
