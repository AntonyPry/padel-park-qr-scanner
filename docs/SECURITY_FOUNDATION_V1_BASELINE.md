## 1. Evidence method and classification

This document inventories the security-relevant application state at the exact base above. It was produced from tracked source, tracked OpenAPI/route declarations, tracked environment examples, package manifests/lockfiles and operational documentation. Real `.env` files, production rows, credentials, hashes, tokens, private keys and personal data were not read or recorded.

Every finding uses one of these labels:

- **[V] Verified** — directly verified in tracked code or a generated contract at the exact base. The adjacent link identifies the file plus function, class or route and current line range.
- **[R] User-reported** — infrastructure or deployment state recorded in project documentation, but not independently observed by SEC-A1.
- **[E] Evidence required** — an application-to-infrastructure contract or operational claim that requires dated evidence from the DevOps mentor/owner before the dependent capability can be accepted.

Normative interpretation is deliberately narrow and uses primary guidance current on the evidence date: [OWASP Password Storage](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html), [Session Management](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html), [CSRF Prevention](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html), [Authentication](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html), [MFA](https://cheatsheetseries.owasp.org/cheatsheets/Multifactor_Authentication_Cheat_Sheet.html), [Forgot Password](https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html), [Logging](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html), [HTTP Headers](https://cheatsheetseries.owasp.org/cheatsheets/HTTP_Headers_Cheat_Sheet.html), [Cryptographic Storage](https://cheatsheetseries.owasp.org/cheatsheets/Cryptographic_Storage_Cheat_Sheet.html), the [Express proxy guide](https://expressjs.com/en/guide/behind-proxies.html), [npm audit documentation](https://docs.npmjs.com/cli/v11/commands/npm-audit/) and [GitHub security features](https://docs.github.com/en/code-security/getting-started/github-security-features). Recommendations below are Setly-specific conclusions from those sources, not a generic security checklist.

### 3.2 Public-entry abuse budget

**[V]** The repository contains no normal auth logout/recovery/MFA route and no rate-limit middleware/package at this base. The public attack budget therefore concentrates on login/bootstrap/operator-session/activation, five provider ingresses, seven worker operations, plus health/OpenAPI enumeration. The application should preserve generic credential errors already used by normal login while A4 adds per-account and verified-source throttling. ## 7. Dependency, rollout, rollback and no-lockout matrix

### 7.1 Required dependency graph

```text
A2 password hashing
  -> A3 operator verifier
  -> A5 opaque server sessions
  -> A6 HttpOnly cookies + CSRF/origin
  -> A12 recovery + revoke-all
  -> A7 MFA + step-up

A4 throttling + A8 browser/HTTP containment = cross-cutting containment boundaries
A9 = extension/permanent gate for existing tenant enforcement
A10 = extension of AuditLog/provider redaction after auth/session event semantics stabilize
A11 = blocked until accepted classification/key lifecycle and I8/I10/I11 evidence
A13 = disjoint later supply-chain slice; no dependency churn during the auth chain
```

A4 and A8 may be developed beside the sequential chain, but each affected public/privileged surface must have its containment control accepted before enforcement expands. A10 event schemas should follow stable A5/A6/A12/A7 semantics so the audit contract does not churn or miss transitions.

## 8. Application ↔ infrastructure contracts

SEC-A1 performs no infrastructure action. The following is the minimum evidence contract for the linked infrastructure slices.

### 8.1 I1 backup/restore coverage

The application contract is the installation-wide manifest and restore rehearsal already tracked. The DevOps mentor must provide **[E]** dated, secret-free evidence containing:

1. Backup artifact identifiers/timestamps and the exact application SHA; database consistency window plus every mandatory tenant/file artifact from `tenant-backup-manifest`.
2. Encryption mechanism and credential/key **class**, custodian/owner, separation from the backup data and recovery process — never the key/value itself.
3. Off-host storage class, access boundary, schedule, retention and immutable/versioned protection; actual and target RPO/RTO.
4. Manifest/checksum verification result, missing/orphan result and worker derived-data rebuild policy.
5. An isolated restore target, restore date, restored schema/application SHA, application/tenant/file smoke result and measured restore duration.
6. Proof that provider ciphertext and the separately protected key lifecycle can both be recovered; a DB-only restore that cannot decrypt required data is not accepted.
7. Owner sign-off for deletion/retention and a rollback decision record. No production rows, filenames containing personal data, credentials or secret material belong in the evidence attached to this baseline.

NIST primary recovery guidance supports maintaining and exercising recovery plans rather than treating backup creation as proof: [SP 800-34 Rev.1](https://csrc.nist.gov/pubs/sp/800/34/r1/upd1/final), [SP 800-184](https://csrc.nist.gov/pubs/sp/800/184/final) and [NIST data-loss protection note](https://csrc.nist.gov/pubs/other/2020/04/24/protecting-data-from-ransomware-and-other-data-los/final).

### 8.2 I4 proxy input, structured events, health and alerts

#### Real client IP / proxy trust input for A4

**[V]** Provider diagnostics use `req.ip`, while Express has no explicit `trust proxy` configuration at this base. [`provider-ingress.js: ingress rejects` (`L44-L105`)](../server/src/middleware/provider-ingress.js#L44-L105), [`app.js: createApp` (`L40-L97`)](../server/src/app.js#L40-L97)

Before A4 trusts source IP, DevOps must provide **[E]**:

- the exact hop topology from client/CDN/load balancer/Nginx to Node and the trusted hop/CIDR decision;
- proof that Node is not directly Internet-reachable and that the last trusted proxy overwrites, rather than appends untrusted, forwarding headers as required by the chosen Express `trust proxy` mode;
- controlled-request evidence showing two real clients produce distinct effective sources, while a client-supplied `X-Forwarded-For` cannot choose another rate-limit bucket;
- IPv4/IPv6/NAT behavior, privacy-preserving retention and the fallback when the trusted proxy signal is absent.

Setting `trust proxy=true` without topology proof is explicitly rejected; the official [Express guidance](https://expressjs.com/en/guide/behind-proxies.html) makes the setting dependent on the actual proxy path.

#### Structured redacted event output

A10 should extend `AuditLog` and provider diagnostics with a versioned, allowlisted event envelope. The infrastructure contract may accept: timestamp, event type/version, severity, outcome/reason code, request/correlation ID, route/surface, pseudonymous actor/session/source identifiers and authoritative tenant scope only when allowed. It must reject raw passwords, tokens, session IDs, cookies, authorization headers, provider credentials, encryption keys, database connection strings, raw request bodies, raw recovery/MFA values and unnecessary personal data.

Required event families: login success/failure/throttle; operator session create/revalidate/revoke/deny; session create/replay/revoke/expiry; recovery/MFA request/change/failure; provider/worker rejection; tenant-context/authorization denial; audit-pipeline failure; degraded health; backup/restore failure. OWASP’s [Logging Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html) is the normative redaction/event baseline.

#### Health/alert acceptance

**[V]** `/health` reports aggregate DB/cache/tenant-foundation state and returns degraded status; the tracked health-check script fails when the endpoint is not healthy. [`routes/index.js: /health` (`L59-L103`)](../server/src/routes/index.js#L59-L103), [`health-check.js` (`L1-L18`)](../server/scripts/health-check.js#L1-L18)

DevOps must provide **[E]** for each accepted alert: owning role, environment, signal/query, threshold/window, deduplication, notification channel, acknowledgement/escalation deadline, maintenance suppression and a dated synthetic test with receipt/acknowledgement. At minimum cover repeated normal/operator/MFA/recovery failures, revoked/replayed sessions, provider/worker authentication rejects, tenant-denial spikes, audit delivery failure, health degradation and backup/restore failure. This document does not select or configure a monitoring product.

## 10. Non-goals and rejected duplicate subsystems

SEC-A1 does not change code or operate infrastructure. It does not conduct penetration testing, read production data/secrets, rotate credentials, change GitHub settings, select vendors, set retention policy, enable MFA, encrypt fields, change dependencies or assert compliance certification.

The following duplicate designs are explicitly rejected:

- no replacement auth service beside `auth.service.js`; A2/A5/A6/A12 evolve its contract;
- no second operator auth/session stack beside `installation-operator-auth.service.js` and `InstallationOperatorSession`;
- no client-supplied or parallel tenant resolver beside route scope declarations, authoritative tenant context, capabilities and final enforcement audits;
- no second audit/log product model beside `AuditLog`, installation audit and provider diagnostics/redaction;
- no second cryptographic envelope/fingerprint format beside provider AES-GCM/HMAC; A11 generalizes the accepted primitive and key lifecycle only after classification;
- no reuse of owner activation as account recovery; its high-entropy/digest/single-use properties are patterns, while authority and lifecycle remain separate;
- no application-side substitute for I1 backup operations or I4 proxy/monitoring ownership.

## 13. Residual baseline risks

This baseline reduces uncertainty; it does not reduce runtime exposure. The highest verified risks awaiting later slices are the JavaScript-readable normal/operator credentials, stateless normal sessions without per-device revocation, plaintext operator password source, lack of abuse throttling, permissive CORS/header posture, legacy Evotor’s optional secret, incomplete security-event/redaction coverage and unverified key/backup/proxy/alert operations. Reported ops-domain/TLS state and all I1/I4 evidence remain unverified until the named owners provide dated artifacts.

Any runtime/config/schema/dependency/workflow change after the exact base invalidates only the affected findings, but the baseline must be refreshed before that changed surface is promoted.
