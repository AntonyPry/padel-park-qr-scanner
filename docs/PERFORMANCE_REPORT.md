# Performance First

Дата замера: 2026-05-24.

Цель: сначала ускорить текущую CRM и доказать эффект цифрами, потом возвращаться к новым продуктовым спринтам.

## Условия замера

- Backend: локально на `http://127.0.0.1:3004`.
- База: локальная MySQL.
- Нагрузочные данные:
  - клиентов: 5 048;
  - визитов: 40 040;

  - позиций чеков: 23 881;
  - задач обзвона: 38;
  - клиентов в задачах: 15 033;
  - баз клиентов: 15.

- Артефакты:

  - `outputs/performance/2026-05-24/after-final.json`;

- Временные `[perf]`-данные после замеров удалены из локальной базы.

## Что было найдено

Главная проблема была не в одном медленном запросе, а в каскадном эффекте.

`GET /api/call-tasks?status=active` и `GET /api/call-tasks/report?status=active` запускали синхронизацию dynamic-задач прямо во время чтения списка/отчета. На большой базе это забивало pool подключений Sequelize, после чего даже другие разделы начинали уходить в timeout.

Дополнительно:

- не хватало составных индексов под горячие запросы;
- стартовый frontend bundle грузил все страницы сразу;
- аудит-лог отдавал тяжелую `metadata` в списке, хотя UI ее не использовал;
- аналитика визитов считала метрики в Node.js после загрузки всех визитов с клиентами.

## Frontend

До оптимизации основной JS:

- `index-*.js`: 1 272 KB;
- gzip: 350 KB.

После route-level lazy loading:

- стартовый `index-*.js`: 208.25 KB;
- gzip: 65.83 KB.

Страницы теперь догружаются по маршруту: входы, клиенты, финансы, задачи, базы, тренерский кабинет и остальные тяжелые разделы не попадают целиком в первый экран.

После финального build выполнен UI-smoke в Chrome по ключевым разделам:

- монитор входов;
- клиенты;
- базы клиентов;
- задачи обзвона;
- финансы;
- аналитика посещений;
- журнал действий.

Console `warn/error` пустой. Скриншоты сохранены в `outputs/qa/2026-05-24-performance/`.

## Что изменено технически

- Добавлена performance-миграция с индексами под горячие таблицы:
  - `Users`;
  - `Visits`;
  - `Receipts`;
  - `ReceiptItems`;
  - `Finances`;
  - `Shifts`;
  - `ClientBases`;
  - `CallTasks`;
  - `CallTaskClients`;
  - `CallTaskAttempts`;
  - `AuditLogs`;
  - `ScannerEvents`;
  - `TrainingNotes`.
- `GET /api/call-tasks` больше не запускает синхронизацию dynamic-задач.
- `GET /api/call-tasks/report` больше не запускает синхронизацию dynamic-задач.
- Синхронизация dynamic-задачи остается явным действием и выполняется в detail/sync-сценариях.
- Страницы frontend переведены на `React.lazy`.
- Диаграммы Recharts получили стабильные размеры, чтобы не шуметь warning-ами в консоли при smoke-проверках.
- Аудит-лог в списке больше не отдает `metadata`.
- Аналитика визитов считает основные метрики SQL-агрегациями.
- Добавлен benchmark tool:
  - `scripts/performance-benchmark.js`.
- Добавлен seed нагрузочных данных:
  - `scripts/seed-performance-data.js`.
- Backend добавляет header:
  - `X-Response-Time-Ms`.
- Медленные API можно логировать через:
  - `SLOW_API_LOG_MS`.

## Что осталось как следующий уровень

Сейчас главные тормоза закрыты. Дальнейшее ускорение уже лучше делать не вслепую, а когда будет production-статистика:

- материализованные `ClientStats`, если клиентов/визитов станет сильно больше;
- отдельный background runner для dynamic-задач;
- кеширование справочников;
- Redis/queue для фоновых задач;
- агрегированные таблицы для P&L и мотивации на очень больших объемах;
- отдельный APM/логирование p95 на production.
